Interactive guide to exploring the 12-layer architecture specification


This guide helps you explore the spec structure with optimized queries for clean visualizations.

  • 354 SpecNode types - Element type definitions

  • 252 SpecRelationship types - Allowed relationships

  • 47 Predicates - Semantic relationship catalog

  • 12 Layers - Architecture layers

Tip: Click any query below to run it instantly!

Quick Verification

Verify the data loaded correctly


Count SpecNodes:

MATCH (n:SpecNode) RETURN COUNT(n) AS specNodes;

Count Layers:

MATCH (n:Layer) RETURN COUNT(n) AS layers;

Count Predicates:

MATCH (n:Predicate) RETURN COUNT(n) AS predicates;

Expected: 354 spec nodes, 12 layers, 47 predicates

The 12 Layers

Explore all layers with their metadata


MATCH (l:Layer)
OPTIONAL MATCH (sn:SpecNode)-[:BELONGS_TO_LAYER]->(l)
RETURN l.number AS layer,
       l.name AS name,
       COUNT(sn) AS nodeTypes,
       l.standard AS standard
ORDER BY l.number;

Tip: Use the right sidebar filters to show/hide node types!

Clean Graph Visualization

The key to clean graphs: limit what you query!


Show only API layer nodes and their layer:

MATCH (api:SpecNode_api)-[:BELONGS_TO_LAYER]->(l:Layer)
RETURN api, l;

Simple cross-layer view:

MATCH (source:SpecNode_api)
MATCH (source)-[:BELONGS_TO_LAYER]->(l1:Layer)
WITH source, l1 LIMIT 5
MATCH (sr:SpecRelationship)-[:HAS_SOURCE]->(source)
MATCH (sr)-[:HAS_TARGET]->(target:SpecNode)
MATCH (target)-[:BELONGS_TO_LAYER]->(l2:Layer)
RETURN source, target, l1, l2;

Focus on One Predicate

Viewing one relationship type at a time creates clearer graphs


"composes" relationships:

MATCH (sr:SpecRelationship {predicate: 'composes'})
MATCH (sr)-[:HAS_SOURCE]->(source:SpecNode)
MATCH (sr)-[:HAS_TARGET]->(target:SpecNode)
RETURN source, target
LIMIT 15;

Try other predicates: influence, realizes, depends-on, references, serves

Explore by Layer

Focus on one layer at a time


API Layer types:

MATCH (sn:SpecNode)-[:BELONGS_TO_LAYER]->(l:Layer {id: 'api'})
RETURN sn.type AS elementType,
       sn.title AS title,
       sn.description AS description
ORDER BY sn.type;

Try other layers: motivation, business, security, data-model, data-store

Cross-Layer Patterns

See which layers reference other layers


MATCH (sr:SpecRelationship)-[:HAS_SOURCE]->(source:SpecNode)-[:BELONGS_TO_LAYER]->(l1:Layer)
MATCH (sr)-[:HAS_TARGET]->(target:SpecNode)-[:BELONGS_TO_LAYER]->(l2:Layer)
WHERE l1.id <> l2.id
RETURN l1.number AS fromLayer,
       l1.name AS fromName,
       l2.number AS toLayer,
       l2.name AS toName,
       COUNT(sr) AS relationshipCount
ORDER BY relationshipCount DESC;

Understanding Predicates

Explore semantic relationship types


Most used predicates:

MATCH (sr:SpecRelationship)-[:USES_PREDICATE]->(p:Predicate)
RETURN p.predicate AS predicate,
       p.category AS category,
       p.description AS description,
       COUNT(sr) AS usageCount
ORDER BY usageCount DESC
LIMIT 10;

Required Attributes

Find validation constraints


Types with most required fields:

MATCH (sn:SpecNode)
WHERE sn.required_attribute_count > 0
RETURN sn.spec_node_id AS nodeType,
       sn.required_attribute_count AS requiredFields,
       sn.total_attribute_count AS totalFields
ORDER BY requiredFields DESC
LIMIT 15;

Attributes for api.endpoint:

MATCH (sn:SpecNode {spec_node_id: 'api.endpoint'})-[r:HAS_ATTRIBUTE]->(attr:AttributeSpec)
RETURN attr.name, attr.type, attr.required, attr.description
ORDER BY r.order;

Explore Around a Node

Focus on what connects to a specific element type


What can an API endpoint connect to?

MATCH (endpoint:SpecNode {spec_node_id: 'api.endpoint'})
OPTIONAL MATCH (sr:SpecRelationship)-[:HAS_SOURCE]->(endpoint)
OPTIONAL MATCH (sr)-[:HAS_TARGET]->(target:SpecNode)
RETURN endpoint, target
LIMIT 20;

Try: motivation.goal, business.service, data-store.table

Find Hub Types

Which types are most connected?


MATCH (sn:SpecNode)
OPTIONAL MATCH (sn)<-[:HAS_SOURCE]-(outgoing:SpecRelationship)
OPTIONAL MATCH (sn)<-[:HAS_TARGET]-(incoming:SpecRelationship)
WITH sn,
     COUNT(DISTINCT outgoing) AS canReferTo,
     COUNT(DISTINCT incoming) AS canBeReferencedBy
RETURN sn.spec_node_id AS nodeType,
       canReferTo,
       canBeReferencedBy,
       canReferTo + canBeReferencedBy AS total
ORDER BY total DESC
LIMIT 10;

Visualization Tips


For cleaner graph views:

  • Use node label filters (right sidebar) - click to show/hide types

  • Drag nodes manually to arrange them

  • Use LIMIT generously - start with 10-15 nodes

  • Focus on one predicate at a time

  • Use zoom-to-fit button (bottom right)

Use Cases


1. Impact Analysis:

MATCH (sn:SpecNode {spec_node_id: 'api.endpoint'})
MATCH (sr:SpecRelationship)-[:HAS_SOURCE|HAS_TARGET]->(sn)
RETURN sr.id AS affectedRelationship,
       sr.predicate AS relationship;

2. Find string validations:

MATCH (sn:SpecNode)-[:HAS_ATTRIBUTE]->(attr:AttributeSpec)
WHERE attr.required = true AND attr.type = 'string'
RETURN sn.spec_node_id, attr.name, attr.description
ORDER BY sn.spec_node_id;
